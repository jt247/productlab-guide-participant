The Mental Model

Before we touch any tool — we align on how to think.

01

The Leverage Shift

Why you're more powerful than you think

The engineering bottleneck is a myth you've accepted as fixed reality. AI has handed non-engineers a superpower — most people aren't using it. That changes today.
Question If you could ship a product this weekend — what would you build?
  • You don't need to write code to ship a real product in 2025
  • The bottleneck is no longer engineering — it's imagination and execution speed
  • Today you leave as a builder, not just a thinker
02

The Spectrum: No-Code → AI-Code

Pick your lane — or use all three

There is no single right tool. There are three zones — and you choose the right one for the right job.
Zone 1
No-Code
Lovable · Adalo
Describe it. AI builds it.
Zero syntax required.
Zone 2
Low-Code
AI Studio · Stitch · Firebase
Guide and configure.
Some logic, more control.
Zone 3
AI-Code
Claude Code · Cursor · Windsurf
You direct, AI writes.
You review and approve.
Tip You don't need to choose one zone forever. The best builders use all three depending on what they're trying to achieve and how fast they need to move.
03

What "Shipped" Actually Means

The only output that counts

  • A live URL someone else can open and use — that is a shipped product
  • Minimum viable: one problem solved, one user action possible, one data point captured
  • The startup loop: Build → Measure → Learn → Repeat
  • The failure mode: over-engineering before validating a single assumption
Tip Your mantra for today: "Live beats perfect." A working URL that does one thing is worth more than a 40-slide deck about a product that doesn't exist.

Phase 2 · 1:30 – 2:00 PM

Idea to Architecture

How to think before you build. Turn a fuzzy idea into a concrete, buildable spec — in 10 minutes.

06

The OCP Framework

One Customer Problem — 12 decisions, zero waffling

Before you open any tool, you complete the OCP. Twelve "ones" — each forces a single, clear decision. When all twelve align, you have something worth building.
01
Customer Problem
The single most painful problem your target user faces today
02
Solution
Your product's answer — one sentence, no ands, no buts
03
Value Proposition
Why a user picks you over doing nothing — the tangible outcome
04
Killer Feature
The one capability that makes users try — and tell others
05
Monetisation Model
How this makes money — subscription, one-time, marketplace, freemium
06
Core User Segment
The one type of person you build for first — be precise, not broad
07
Key Strategy
Your primary growth lever — virality, SEO, partnerships, community, outbound
08
Key Channel
Where your users already live — LinkedIn, WhatsApp, X, communities, email
09
Key Metric
The one number that tells you the product is working
10
Unfair Advantage
What you have that a competitor cannot easily copy
11
Retention Hook
Why users come back — habit, lock-in, community, or ongoing value
12
Growth Loop
How the product grows itself — referrals, SEO, viral sharing, network effects
Task 10 minutes. Complete your OCP canvas for the idea you want to build today. One answer per box. No paragraphs — one sentence maximum. You'll use this in Lab A.
Tip Before Lab A starts, go to Product Nerve AI — productnerve.com and run your completed OCP through it. It's a Venture Operating System built to stress-test founder ideas. The validation output will sharpen every build decision you make today.
Tip If you can't answer all twelve in 10 minutes, the idea needs more thinking — not more building. That's the whole point of the OCP.
07

The SPARK Framework

How to write prompts that actually build what you want

The quality of what you build with AI = the quality of your prompt. SPARK is your structure every time you open Lovable, Claude Code, or AI Studio.
S
Scope
What are you building?
App? Landing page? Dashboard? API? Be specific about the type of output.
P
Persona
Who is the user?
What do they need to accomplish? What problem are they solving?
A
Actions
What can users do?
Name the 3 key things a user must be able to do in this product.
R
Result
What happens next?
What does the system do when the user completes each action?
K
Keep Out
What should this NOT do?
Scope limits are as important as scope itself. Name what's excluded.
Task Write your SPARK prompt for the product you want to build in Lab A today. You'll paste this directly into Lovable when we start.

Phase 3 · 2:00 – 2:30 PM

The Invisible Stack

The infrastructure that determines whether your product survives real users.

09

Infrastructure Is the Product

What keeps your product alive after launch

The product that dies after 100 users wasn't ready — the infrastructure was missing. Today's tools make production-ready the default, not a milestone you reach later.
  • Database: Where your data lives — Supabase or Firebase. Real databases, free to start.
  • Auth: How users log in — email, Google, magic links. Built into Supabase and Firebase.
  • Email: Confirmations, resets, receipts — Resend or Zeptomail. Set it up once.
  • Monitoring: Sentry catches errors. Uptime Robot alerts you when your product goes down.
  • Analytics: PostHog tracks behaviour. GA4 + GTM manage all your tracking tags.
  • Payments: Stripe for global, Paystack for African markets. Add it from day one.
Tip A founder's payment page broke for 48 hours — they lost 40 signups before noticing. Sentry + Uptime Robot cost nothing to set up and would have caught it in minutes.
10

Supabase in 60 Seconds

Your backend — without a backend engineer

  • Database: Postgres tables — looks like Airtable. You already know how to use this.
  • Auth: Email/password, Google OAuth, magic links — zero extra code needed
  • Storage: File uploads (images, PDFs, documents) with access controls
  • Realtime: Live updates across all users — no WebSocket code required
Tip Lovable connects to Supabase in one click. Your data lives in a real database from the very first prompt.
11

Analytics, Marketing & Payments

Track everything. Charge from day one.

  • PostHog / GA4: Track activation (did the user do the core action?) and retention (did they come back?)
  • GTM: Manages all your tracking tags — Meta Pixel, GA4, custom events — from one place, without touching code every time
  • Meta Pixel + CAPI: Browser + server-side tracking. Closes the iOS attribution gap for all paid social campaigns
  • UTM parameters: Tag every link — LinkedIn, X, WhatsApp, email — so you know exactly which channel drove each signup
  • Stripe / Paystack: Add payment on day one. Even if free, having the payment infrastructure signals serious intent
Tip Don't measure everything. Measure the one number that tells you users are getting value from your product. For most early-stage products that's activation — did the user complete the core action?
12

Deploy: Localhost to Live in 3 Minutes

The moment you hand someone a URL — you become a founder

  • Lovable deploys to Vercel by default — one click, live URL, done
  • Custom domain: ~$10/year on Namecheap → point DNS to Vercel → live in 5 minutes
  • Backend services go on Railway — same push-to-deploy simplicity
  • GitHub is your safety net — every change tracked, every deploy triggered automatically
Tip The moment you hand someone your URL — that's when you become a founder. You're about to do that. Labs start in 10 minutes.

Lab A · 2:40 – 4:10 PM · 90 minutes
Lovable: AI Startup Directory
We build the platform together — then you list your idea on it.
What We're Building
  • A live, public AI-Assisted Startup Directory — a real platform where today's ideas are submitted, browsed, and spotlighted
  • Homepage with a browsable card grid — filterable by category, tool used, and stage
  • Submit page — name, idea title, problem solved, tool used, live URL, one-liner pitch
  • Each submission gets its own public detail page showing the full OCP breakdown
  • A "Featured by JT" spotlight badge — awarded post-workshop to the best submissions
  • Supabase backend — all ideas stored in a real database, visible in real time
  • Resend confirmation — every submitter gets a branded email when their idea goes live
  • PostHog analytics — tracking visits and submissions from day one
  • Deployed to Vercel — live and shareable before the session ends
Step-by-Step
  • 0:00SPARK prompt — write and run the Lovable prompt for the full directory. Watch the first version appear.
  • 0:08Iterate — refine the card layout, fix the submission form fields to match the OCP structure.
  • 0:25Connect Supabase — create the ideas table. Enable admin auth.
  • 0:40Wire Resend — submission confirmation email fires automatically on every new entry.
  • 0:50Add PostHog snippet. Deploy to Vercel. Test the full submission flow.
  • 1:00Open submissions — every participant fills in and submits their OCP idea live.
  • 1:20Browse together. JT announces which ideas will receive the post-workshop spotlight.
Task Before this lab starts: have your completed OCP canvas ready. You'll be submitting it to the live directory before the session ends.
Tip Don't wait for the product to be "perfect" before submitting your idea. A rough OCP that's honest is more valuable than a polished one that's vague.
⏱ 90 min · No-code · All skill levels
Lab B · 4:20 – 5:50 PM · 90 minutes
Google AI Studio + Stitch + Firebase
Design-first, AI-powered — three Google tools, one complete product.
What We're Building
  • Smart Feedback Collector — users submit product feedback, Gemini AI categorises it by theme and sentiment, results render live in a dashboard
  • Submission screen: clean form UI built and exported from Stitch — real code, not a mockup
  • Gemini AI layer: feedback → Gemini API → returns category tag + sentiment score
  • Firebase Firestore: all submissions + AI results stored and streamed in real time
  • Dashboard: live feed of categorised feedback with visual sentiment breakdown
  • Google Antigravity: introduced as an alternate build path to explore after the workshop
Step-by-Step
  • 0:00Define the 3 core screens: Submit Feedback, Thank You, Dashboard.
  • 0:10Use Stitch to generate all three screens. Export the code. Review the structure.
  • 0:30Open Google AI Studio — create a Gemini prompt that returns category + sentiment as JSON. Test with live examples.
  • 0:50Wire Gemini into the form: submit → call Gemini → store result in Firebase Firestore.
  • 1:05Connect the dashboard to Firebase — real-time listener renders new submissions as they arrive.
  • 1:20Quick Antigravity intro — what it is and where it fits as an alternative.
  • 1:25Test the full flow end-to-end. Prep 2-min demo.
Question Where in your current workflow could an AI categorisation layer like this save you the most time?
Tip Everything built in this lab is real code. Not a wireframe, not a prototype — deployable code generated by Stitch and powered by Gemini. You can take this and extend it after the workshop.

Lab C · 6:00 – 7:30 PM · 90 minutes
Claude Code + Adalo
Two builds in one lab — a web app via Claude Code Desktop, and a mobile app via Adalo.
Build 1 — GTM Campaign Planner (Claude Code · 50 min)
  • A web app where you input your product details and receive a structured, shareable go-to-market campaign brief
  • Inputs: product name, target audience, key message, channels, campaign goal
  • Output: structured GTM brief — channel plan, messaging hierarchy, CTAs, success metrics — formatted as a downloadable doc
  • Supabase stores all campaigns. Resend emails the finished brief to the user automatically.
  • Built entirely through the Claude Desktop App — prompting every step, no terminal required
  • Deployed to Vercel via GitHub — live URL in under 50 minutes
Build 1 — Step-by-Step
  • 0:00Open Claude Desktop App. OCP the GTM Planner. Prompt Claude to scaffold the full project.
  • 0:08Review file structure. Prompt auth: "Add Supabase email login and a protected dashboard."
  • 0:20Prompt the core form: "Build a GTM brief form — product name, audience, key message, channels, goal. Save to Supabase."
  • 0:35Prompt the output: "Generate a structured GTM doc from the form. Make it downloadable. Email it via Resend."
  • 0:45Push to GitHub → deploy to Vercel. Live URL. Test the full flow.
Build 2 — Booking & Appointment App (Adalo · 35 min)
  • A native mobile app for iOS & Android — service listing, date/time picker, booking confirmation, appointment history
  • Built in Adalo's visual editor — no code written at all
  • Embeddable booking widget: Adalo generates a web snippet you can drop into any website, Notion page, or platform
  • Use case: any service business — consultant, coach, clinic — can take bookings without a custom dev build
Build 2 — Step-by-Step
  • 0:50Create Adalo project. Set up collections: Services, Users, Bookings. Define relationships.
  • 1:00Build core screens: Home (service list), Book (date/time + confirm), My Bookings. Link all screens.
  • 1:15Enable auth. Test the full login and booking flow on mobile preview.
  • 1:20Show the web embed snippet. Paste into a live page — demonstrate the booking flow embedded in a browser.
  • 1:25Recap both builds. Prep 2-min demo.
Question Where in your current work could a structured GTM brief like this replace a document that currently takes hours to write?
Tip You never wrote a single line of code in Build 1. You directed — Claude executed. Every prompt you wrote was a spec. That's the new workflow.
Tip Adalo's embed feature means your mobile app is also a widget. One build, multiple surfaces. Drop it into your website, your client portal, your community — anywhere.
